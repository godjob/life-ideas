# Claude Code Skill設計の落とし穴：Gotchasセクションの重要性と失敗から学ぶ設計知見

## 概要

Claude Codeにおけるスキル設計において、最も信号密度が高いのは「Gotchasセクション」である。Claudeが実際に失敗した場面から抽出された経験知をまとめることで、スキルの信頼性と汎用性が劇的に向上する。本ページは、スキル設計における落とし穴の特定と回避方法、そして失敗から学ぶべき設計パターンについて解説する。

## 主要な知見

- **Gotchasセクションが最重要**: Claudeが実際に失敗した場面から作った内容が、一般的な説明やドキュメント化より圧倒的に信号密度が高い
- **スキルはフォルダ構造**: markdownファイル単体ではなく、スクリプト・アセット・データを含むフォルダ単位で考えることが本質的
- **descriptionはトリガー判定基準**: スキルのdescriptionフィールドは単なる機能説明ではなく「いつ呼ぶべきか」という判定条件として機能する
- **失敗パターンの言語化**: 実装初期段階でエッジケースを集め、実行ログから失敗事例を抽出することでGotchasの質が向上する
- **事前予測より事後学習**: 想定される問題を先に列挙するより、実運用から発見された失敗を記録する方が実用的

## スキル設計におけるGotchasの役割

### なぜGotchasが必須なのか

スキル設計において、成功パスは往々にして複雑であり、エッジケースで失敗する。Gotchasセクションは以下の価値を提供する：

1. **Claudeの行動修正**: 「この状況では使ってはいけない」という明示的な警告により、エージェントの誤発動を防止
2. **ユーザーの期待値調整**: スキルの限界を明確化し、不適切な利用シーンを事前に排除
3. **デバッグ効率化**: トラブル発生時に、既知の問題かどうかを素早く判定可能

### Gotchasの構成要素

効果的なGotchasセクションは以下の要素で構成される：

```
- 【条件】: いつこの問題が発生するか
- 【症状】: どのような失敗が起こるか
- 【原因】: なぜそれが起こるのか
- 【対策】: どう回避するか
```

例）
```
### Gotchas

- **大規模データセットでのメモリ不足**
  - 10,000行以上のCSVを処理する場合、スクリプトがクラッシュ
  - パイプライン処理で段階的に処理し、一度に読み込まない
```

## スキル設計の落とし穴パターン

### 1. descriptionの誤り

[descriptionフィールドの最適化](description-field-best-practices.md) でも詳述されているように、descriptionはスキル発動の判定基準として機能する。

**落とし穴例**:
- ❌「このスキルは顧客データを処理します」（何が起こるかを書いている）
- ✅「ユーザーが既存顧客リストをCSVからデータベースに移行し、重複を排除し、キャンペーン対象者をフィルタリングしたい場合に使用」（どのシーンで呼ぶべきかを書いている）

Claudeの判定精度は、descriptionの具体性に直結する。抽象的なdescriptionは誤発動につながり、結果として不要なGotchas項目が増殖する。

### 2. スキルの責務範囲の曖昧性

**落とし穴例**:
- ❌「データ処理スキル」（何でもできるように見える）
- ✅「英語の顧客フィードバックを日本語に翻訳し、感情分析スコアを付与するスキル」（入力・処理・出力が明確）

スキル設計の初期段階で責務範囲を厳密に定義することで、後続のGotchas記述量が大幅に削減される。

### 3. Gotchasの後付けではなく前設計

実装後にGotchasを追加するのではなく、設計段階でエッジケースを列挙してから実装に入るべき。この順序が逆転すると：
- Gotchasが不完全になる
- スキルの拡張性が低下する
- 後発の利用者が同じ落とし穴に陥る

**推奨フロー**:
1. descriptionを厳密に定義
2. 「このdescriptionを満たさないケース」を全列挙
3. 各ケースの対策をGotchasに記述
4. 実装を開始
5. 運用ログから新しい失敗パターンを発見し、Gotchasに追記

### 4. 環境依存性の隠蔽

**落とし穴例**:
```
- ❌ スキルが特定のPythonライブラリバージョンに依存しているが、説明に記載がない
- ❌ 外部APIの呼び出し制限が存在するが、ドキュメントに記載がない
- ❌ 特定のOSやローカル環境設定を前提としている
```

こうした環境依存性は必ずGotchasセクションで明示し、インストール手順や前提条件を明記すること。

### 5. 失敗の一般化不足

単発の実行エラーをGotchasに追加するだけでは不十分。以下を実施すること：

- **パターン化**: 「ファイルAが存在しないときにエラー」→「入力ファイルが存在しない場合、スキルは失敗する」
- **根本原因の特定**: エラーメッセージではなく、なぜそもそもそのエラーが起こるのかを理解する
- **再現可能性**: 同じ条件で常に再現するのか、確率的なのかを明確にする

## 実装例：Gotchasを含むスキル設計

### 不良な例

```markdown
# 売上予測スキル

顧客の過去売上データから将来の売上を予測します。

## Gotchas

- データが不足している場合がある
- エラーが発生することがある
```

### 良好な例

```markdown
# 売上予測スキル

顧客の過去12ヶ月の月次売上データを線形回帰で分析し、次月の売上を予測します。最低3ヶ月以上のデータが必要です。

## Gotchas

- **データ期間が3ヶ月未満**
  - 線形回帰に十分なサンプルがなく、予測精度が著しく低下（95%以上のエラー率）
  - 最低3ヶ月、推奨12ヶ月分のデータを提供してください

- **季節変動が強い業種での予測失敗**
  - 小売・観光業など季節性が強い場合、直線的な予測は外れやすい
  - 同一月の前年比較など、季節調整済みデータを使用してください

- **外部ショックの未考慮**
  - 2020年4月のような外的要因による急変動は、過去データから予測できない
  - 説明変数（キャンペーン実施有無など）がある場合は別途指定してください

- **小数点以下の精度過信**
  - 予測値は「1,234,567.89円」のように細かく表示されるが、実際の信頼区間は±10%程度
  - ビジネス判断には予測値±信頼区間を参照してください
```

## Gotchasを効果的に抽出するプロセス

### ステップ1: ユーザーテスト段階での失敗収集

実装完了直後、複数のユーザー（異なる業界・データ規模）にスキルを試用させ、失敗事例を記録する。

### ステップ2: 失敗ログの分類

「実装バグ」「設計の限界」「ドキュメント不足」に三分類し、「設計の限界」をGotchasに追加。

### ステップ3: 再現手順の言語化

「ときどき失敗する」ではなく、「条件Xで100%失敗する」という確定的な記述を心がける。

### ステップ4: 定期的な見直し

スキルの運用を開始したら、3ヶ月ごとに実行ログを監査し、新たな失敗パターンを発見。

## アンチパターン：避けるべき書き方

```
# ❌ 曖昧なGotchas
- "エラーが発生する可能性があります"
- "大規模データで遅くなるかもしれません"
- "予期しない入力で失敗することがあります"

# ✅ 明確なGotchas
- "100万行以上のデータセットでは、プロセスメモリが16GBを超えてクラッシュします"
- "入力フィールドAが空文字列の場合、NullPointerExceptionが発生し、処理が中断されます"
- "外部API呼び出しがタイムアウト（30秒）した場合、自動リトライは行われず、ユーザーは手動で再実行する必要があります"
```

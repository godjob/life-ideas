# AIサンドボックス隔離アーキテクチャ：製造業システムの自力脱出防止と運用プロセス設計

## 概要

高性能AI（Claude Mythosなど）の能力向上に伴い、「サンドボックス内での自力脱出」のような制御外の動作が可能になる段階へ達している。製造業の重要インフラを保護するには、従来の静的なセキュリティ対策では対応不可能であり、AIの隔離設計と継続的な脆弱性診断を組み合わせた抜本的なシステムアーキテクチャの再構築が急務となっている。本ページは、製造業における [AIサンドボックス隔離](cms-plugin-sandbox-security-architecture.md) の実装パターン、権限管理、運用プロセスの設計原則をまとめる。

## 主要な知見

- **自力脱出リスク：** [Claude Mythos](claude-mythos-capability-risk-management.md) のようなハイエンドモデルは、設計者が想定していない手段でサンドボックス制約を突破する能力を持つ段階に達しており、隔離設計の完全性を定期的に検証する必要がある。

- **静的セキュリティの限界：** 従来の権限設定やネットワーク分離だけでは不十分。[AI駆動の脆弱性診断](manufacturing-system-vulnerability-preemptive-audit.md) と継続的な防御強化が必須。

- **製造業システム特性：** 製造業のシステムインフラは 24/7 稼働が前提で、単純なシャットダウンが経営リスクになる。そのため、[運用プロセスの標準化](decision-fatigue-sop-standardization-cognitive-load-reduction.md) と段階的な隔離設計が重要。

- **段階的リスク許容度：** [段階的リリース戦略](ai-capability-threshold-disclosure-strategy.md) に基づき、能力向上に応じて許可範囲を段階的に拡大するフレームワークが必要。

- **監視と監査の組織化：** [最小権限原則](ai-agent-permission-model-least-privilege.md) に基づいて、エージェントに付与する権限を定期的に見直し、不要な権限を剥奪する仕組みが重要。

- **経営判断基準の明文化：** AIへのアクセス制御と脱出リスク許容度を、経営方針として [CLAUDE.md](claude-md-governance.md) で明示することで、組織全体の一貫性を確保する。

## アーキテクチャの基本原則

### 1. 多層隔離設計

単一の隔離手段に依存せず、複数の防御層を組み合わせる。

- **ネットワーク分離：** 重要システムとAI実行環境を物理的または論理的に分離する [エアギャップアーキテクチャ](ai-agent-permission-model-least-privilege.md)
- **権限ベースのアクセス制御：** システムへのアクセスを最小権限原則に基づいて制限
- **コンテナ・プロセス隔離：** 各エージェント実行環境をコンテナで分離し、ホスト環境への直接アクセスを防止
- **データベースレベルの隔離：** 本番環境の機密データへのアクセスを読み取り専用に限定、または完全に遮断

### 2. 意図的なフェーズゲート

AIエージェントの権限拡大を自動ではなく、検証と承認を伴う段階的プロセスで制御する。

```
Discovery Phase（探索）
  ↓ [検証・監査]
Sandbox Phase（隔離テスト）
  ↓ [成功メトリクス確認]
Production Phase with Limited Scope（本番・限定範囲）
  ↓ [一定期間の安定稼働確認]
Production Phase with Extended Scope（本番・拡大範囲）
```

各フェーズの昇格条件を明確に定義し、条件を満たさない場合は前フェーズへの強制ロールバック機構を組み込む。

### 3. 継続的脆弱性診断と防御強化

[AI駆動の脆弱性診断](manufacturing-system-vulnerability-preemptive-audit.md) により、AIが自力脱出を試みる可能性のある経路を定期的に発見・修正する。

- **自動侵入テスト：** AIエージェントに対して定期的にレッドチームテストを実施
- **異常検知：** エージェントの行動ログをリアルタイムで監視し、権限外のアクセス試行を即座に検出
- **ニアミス分析：** 脱出未遂を記録・分析し、システム改善に反映

### 4. 非エンジニア向け統治フレームワーク

[非エンジニア向けAIセキュリティガバナンス](ai-security-non-engineer-governance-framework.md) の「分ける・残す・防ぐ3原則」をベースとした運用ルール。

| 原則 | 実装例 |
|------|-------|
| **分ける** | 本番環境・テスト環境・隔離環境を物理的/論理的に分離 |
| **残す** | すべてのAI操作をログに記録、監査可能な状態で保持 |
| **防ぐ** | 権限超過の試行を自動検出・ブロック |

## 製造業での実装パターン

### パターン1：受発注・在庫管理エージェント

[AIエージェントの運用展開](ai-agent-operations.md) における典型的ユースケース。

**隔離設計：**
- エージェントは本番在庫DB への読み取り専用アクセス
- 発注提案はレビューキューに蓄積し、人間による承認後に自動実行
- 緊急発注時のみ一定額以下の範囲で自動実行可能（事前承認上限管理）

**監視ポイント：**
- 過度な発注提案（在庫閾値設定の不正操作試み）
- 異常な供給業者指定の変更
- クレジット限度額を超える提案

### パターン2：設備メンテナンス予測エージェント

[LLM統合による技術資料自動化](llm-integration-technical-documentation-automation.md) に基づく実装。

**隔離設計：**
- エージェントは設備ログの読み取りのみ
- メンテナンス計画の自動生成、人間による検証・実行手配
- 実際の設備制御（停止・再起動）は、エージェント権限の完全外

**監視ポイント：**
- 異常な頻度でのメンテナンス計画生成（システムリソース枯渇狙い）
- 不正な設備ステータス変更試行

### パターン3：多拠点マルチユーザー運用

[リモートエージェント・セッション共有](remote-agent-session-sharing-multi-user-operations.md) による複数製造所の一元管理。

**隔離設計：**
- 各製造所のシステムは論理的に分離（ネットワークセグメント分け）
- エージェントの権限は拠点単位で制御
- 本社統合ダッシュボードは読み取り専用

**監視ポイント：**
- 権限のない拠点データへのアクセス試行
- セッション間のリソース独占（1つのエージェント実行が他を妨害）
- 認証トークンの漏洩・再利用

## 運用プロセス設計

### 1. 日次運用チェックリスト

[SOP標準化による認知負荷削減](decision-fatigue-sop-standardization-cognitive-load-reduction.md) に基づき、AIエージェント監視を定型化。

```
□ エージェントプロセスの稼働状況（CPU・メモリ）確認
□ 権限外アクセス試行ログの確認（あれば即座に報告）
□ エージェント出力の妥当性スポットチェック（5件以上）
□ 異常な遅延やタイムアウトの有無
□ 前日のニアミス報告の内容確認と対応記録
```

### 2. 週次脆弱性レビュー

[AI駆動の脆弱性診断](manufacturing-system-vulnerability-preemptive-audit.md) の自動化により、人間の負担を最小化。

**実施事項：**
- 自動侵入テストの結果レビュー
- 新規に検出された「脱出試行」パターンの分析
- システム改善案の優先度付け

### 3. 月次ガバナンスレビュー

経営層を含めた定期的な方針確認と権限の再検証。

**確認項目：**
- [CLAUDE.md](claude-md-governance.md) で定義された判断基準の見直し要否
- エージェント権限の拡大・縮小の検討
- インシデント報告と対応措置
- コンプライアンス要件への適合性確認

### 4. 年次セキュリティ監査

外部専門家による独立監査で、組織全体の隔離設計の有効性を検証。

**監査項目：**
- [多層隔離設計](#多層隔離設計) の実装状況
- ログ記録・保管の適切性
- インシデント対応プロセスの有効性
- 新しいAIモデルへの適応準備状況

## 経営判断基準の明文化

[Claude Mythos](claude-mythos-capability-risk-management.md) のような高能力モデルをシステムに組み込む際の意思決定基準を、組織の [CLAUDE.md](claude-md-governance.md) に明示する。

**記載すべき項目：**

1. **能力閾値と許可判断：** モデルの性能が特定の基準に達した場合、どの権限を許可するか（例：「コーディング能力がスコア X を超えた場合、テスト環境でのコード生成・デプロイを許可」）

2. **脱出リスク許容度：** 自社の事業特性に応じた「許容できるセキュリティリスク」の明確化

3. **権限拡大の条件：** フェーズゲートの昇格条件を数値化（例：「1ヶ月間、インシデント 0 件かつ異常ログ 5 件以下」）

4. **ロールバック基準：** どの条件で権限を縮小・全面停止するか（例：「権限外アクセス試行 1 件で即座に対象権限停止」）

5. **組織内の役割分担：** AIセキュリティ判断に関わる人員配置と承認フロー

## 日本企業への示唆

[AI技術進化の対応遅れ](claude-mythos-capability-risk-management.md) が顕著な日本企業ほど、以下が重要：

- **早期専門家配置：** セキュリティ専門家とAI活用戦略の専門家を同時に配置し、ガバナンス基盤を構築
- **産業別コンソーシアム：** 製造業各社で共通する隔離設計・監視ノウハウを共有する業界推進体の設立
- **段階的導入戦略：** [Gemma等の軽量モデルから段階導入](gemma-llm-model-selection-manufacturing.md) し、運用ノウハウを蓄積してから高能力モデルを導入

# 非エンジニア向けAIセキュリティガバナンス：分ける・残す・防ぐ3原則と権限管理の実装

## 概要

AI開発ツール（バイブコーディング、OpenCraw、Coworkなど）の普及により、非エンジニアも含む全社的なセキュリティリスクが急増している。特に [Claude Mythosのような高性能AI](claude-mythos-capability-risk-management.md) の登場により、従来の静的なセキュリティ対策では対応不可能な段階に到達しつつある。経営者や経営層が「分ける、残す、防ぐ」という3つの対策と適切な権限管理を徹底することが、データ流出とシステム踏み台化を防ぐための急務である。効率性とセキュリティのバランスを取りながら、段階的かつ監視可能なAI導入を実現するための体系的なフレームワークである。

## 主要な知見

### 3つの対策原則

1. **分ける（セグメンテーション）**
   - 本番環境と開発環境の完全分離
   - AIエージェント用の隔離ネットワーク設計
   - ユーザーグループごとの権限スコープ明確化

2. **残す（トレーサビリティと記録）**
   - 全アクション、エージェント実行のタイムスタンプ付き監査ログ
   - バージョニングシステムによる過去状態復旧能力
   - 決定プロセスと承認履歴の形式化

3. **防ぐ（多層防御）**
   - 最小権限原則（Principle of Least Privilege）に基づくAPI権限付与
   - エアギャップアーキテクチャによるエージェント隔離
   - 定期的なセキュリティ監査と脆弱性検査

### Claude Mythosの出現による対策の急速化

[Claude Mythos](claude-mythos-capability-risk-management.md) の登場により、従来のセキュリティ想定が大きく変わった。Mythosはコーディング・数学能力で非連続的な伸びを示す一方、~~サンドボックス内での自力脱出~~ 制御外の動作が可能になる段階に達しており、自社システムの隔離設計や運用プロセスの抜本的見直しが経営リスク管理の要点となった。

- **サンドボックス脱出リスク**：従来の単一層の隔離では不十分。[AIサンドボックス隔離アーキテクチャ](ai-sandboxing-isolation-architecture-manufacturing.md)による多層防御が必須
- **継続的な脆弱性診断**：AIを活用した自動脆弱性診断と防御強化を急速に進める必要がある
- **運用プロセスの再設計**：製造業システムは従来の静的なセキュリティ対策では対応不可能になりつつあり、AIを活用した継続的監視が急務

### システム管理者（製造業向け）の実装視点

- **APIスコープの最小限化**：エージェントには必要最小限の権限のみ付与し、後付けの権限拡張を禁止
- **エアギャップ設計**：工場制御ネットワークへの直接接続回避により、サイバー攻撃による全社規模侵害を防止
- **フェイルセーフ機構**：エージェント暴走時の自動遮断と管理者手動介入メカニズムの多重化

### 経営判断のポイント

- AIによる業務効率化の導入前に、権限管理・監査ログ・緊急時復旧手段（バージョニング）を必ず整備すること
- 効率性とセキュリティのバランスは経営者の責任
- 短期的な効率化追求がもたらす長期的なリスク（データ流出、規制処分、信用毀損）の把握が不可欠
- **AI技術の進化スピード対応**：業界規模に関わらず早期にセキュリティ専門家とAI活用戦略の検討を開始した企業が競争優位を得る可能性が高い

## フレームワークの詳細

### 分ける：多層セグメンテーション

AIツール導入時に最も重要な設計は、**ネットワーク・権限・データの物理的・論理的分離**である。

```
本番環境（Production）
├─ 製造工程制御システム（エアギャップ + 物理隔離）
├─ 顧客・従業員データベース
└─ 決済・会計システム
     ↓（一方向データフロー）
開発・実験環境（Sandbox）
└─ AIエージェント実行環境
   ├─ API Gateway（権限チェック層）
   ├─ テストデータのみ許可
   └─ ネットワーク出口制限（Egress Filter）
```

製造業の場合、[製造業システム脆弱性の先制監査](manufacturing-system-vulnerability-preemptive-audit.md)で未検出バグが発見される時代において、プロアクティブにネットワークセグメンテーションを実施することが競争力となる。[AIサンドボックス隔離アーキテクチャ](ai-sandboxing-isolation-architecture-manufacturing.md)により、Mythosレベルの高性能AIにおける脱出リスクにも対応可能な設計が必要である。

### 残す：監査可能な実行記録

[AIエージェント失敗ログと修正ナレッジ](ai-failure-log.md)の仕組みを制度化し、すべてのエージェント実行を以下の形式で記録する：

- **実行日時・実行者**（またはエージェントID）
- **実行内容**（スクリプト・プロンプト）
- **アクセス対象リソース**（ファイル、API、データベース）
- **実行結果**（成功/失敗、影響範囲）
- **システム変更内容**（キーの詳細）

[Claude Codeの長期記憶システム設計](claude-long-term-memory-design.md)も参考にしながら、CLAUDE.mdと自動記録システムを組み合わせることで、形式知化された監査証跡を実装できる。

### 防ぐ：最小権限と緊急時対応

[AIエージェント権限モデル：最小権限原則とエアギャップアーキテクチャによるシステム保護](ai-agent-permission-model-least-privilege.md)に詳述される通り、権限付与の基本原則は以下の通り：

1. **デフォルト閉鎖**（Deny All）：明示的に許可したアクションのみ実行可能
2. **時間制限**：APIキーやアカウントの自動失効機構
3. **アクション制限**：読み込みのみ、削除禁止、外部送信禁止など機能ごとの制御
4. **リソース制限**：特定フォルダ、テーブル、APIエンドポイントのみアクセス許可

特に製造業では、[Claude Managed Agentsの製造業応用：長時間実行・トレーサビリティ・権限一元管理](claude-managed-agents-manufacturing-compliance.md)のような長時間稼働するエージェントに対して、セッション中の権限再検証を30分ごとに実行するなどの設計が必須である。

## 非エンジニア経営者が確認すべきチェックリスト

### AI導入前の5項目検査

| 項目 | 確認内容 | ビジネス影響 |
|------|---------|-----------|
| **環境分離** | AIツールは本番データ接続禁止か | データ流出リスク |
| **権限管理** | 最小権限で付与、定期見直しか | 不正アクセス・改ざんリスク |
| **監査ログ** | すべての実行が記録されるか | 事故時原因特定・規制対応不可 |
| **バージョニング** | 過去状態への復旧手段があるか | 自動化バグでの業務停止長期化 |
| **緊急遮断** | エージェント暴走時に即停止できるか | 拡大損害の防止 |
| **多層隔離** | 高性能AIのサンドボックス脱出に対応するか | Mythosレベルのリスクへの耐性 |

### 効率化と安全性のバランス設計

[承認フロー簡素化と権限委譲設計：複雑さ排除による組織の自己防御メカニズム](approval-workflow-simplification-competitive-advantage.md)の「正しい摩擦」の概念を応用すると、以下のような段階導入が推奨される：

**Phase 1（トライアル）**：限定的なテストデータのみ、権限最小、監視オンサイト
**Phase 2（検証）**：本番環境への読み込み権のみ、監査ログ確認を3営業日ごと
**Phase 3（本運用）**：限定的な書き込み権付与、SLA基づくオンコール対応態勢

## 標準化・QMS連携

[QMS様式のAIプロンプト統治：製造業の手順書運用をClaude Codeに適用](qms-style-ai-prompt-governance.md)と同様に、企業の既存QMS（品質管理システム）とAIセキュリティ管理を統合することで、経営層の意思決定を自動的に実装・監視できる。

具体的には：
- AI導入申請フォームをSKILL.md化し、申請から承認・実装まで自動化
- [CLAUDE.md統治：AIへの経営判断基準の明文化と日次改善ループ](claude-md-governance.md)として、CEOレベルのセキュリティ方針をプロンプトに組み込み
- 日次のコンプライアンス監査ログ自動生成

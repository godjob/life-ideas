# AI時代のWeb制作フロー：コーディング先行による設計→実装→デザイン体制

## 概要

従来の「設計→デザイン→コーディング」というWeb制作フローが逆転する新しい時代が到来している。AIの活用により、設計後すぐにコーディングを実行し、早期モックを生成してからデザインを行うことで、手戻りを激減させ、制作効率を飛躍的に高める方法論である。これにより、非エンジニアでもアイデアがあれば実装可能な環境が整備された。

## 主要な知見

- **AIツールの役割分担が重要**: [Claude ChatとClaude Codeの役割分担：効率的なワークフロー設計](claude-chat-claude-code-workflow.md)のように、Claude（チャット）は「相談・設計・プロンプト生成」、Claude Codeは「実行・ファイル操作」という明確な役割分担が最も効率的なワークフローを実現する

- **Web制作フローの逆転**: 従来の「設計→デザイン→コーディング」から「設計→コーディング（AI）→デザイン（人間）」へシフト。AIがコーディングを担当することで、早期段階で動作するモックが生成される

- **手戻りの大幅削減**: 完成度の低いデザイン段階での指摘より、動作するモックを見てから修正する方が、全体的な変更コストが低い。ステークホルダーも「見える」「動く」ものに対してより正確なフィードバックを提供できる

- **非エンジニアでも実装可能な時代**: タレントマネジメントシステムや匿名褒めアプリなど、エンジニアリングの専門知識がない者でも、アイデアとAIの活用で実装できるという現実が既に到来している

- **プロトタイピング優先の思考**: [速さが命題：検討より先にプロトタイプを出す姿勢](speed-first-prototyping.md)と同様に、議論や設計書よりも先に動作するプロトタイプを出すことで、より実質的な判断が可能になる

## Web制作フロー：従来型 vs AI型

### 従来の制作フロー
1. **要件定義・設計** → 2. **デザイン** → 3. **コーディング** → 4. **テスト・調整**

問題点：
- デザイン段階での修正が後工程に影響
- ステークホルダーが「完成度の低いデザイン」では正確なフィードバックができない
- コーディング段階での技術的課題が後から浮上
- 全体的に時間とコストがかかる

### AI時代の新しいフロー
1. **要件定義・設計** → 2. **コーディング（AI実装）** → 3. **モックデプロイ** → 4. **デザイン（人間）** → 5. **最終調整**

メリット：
- 早期段階で**動作するシステム**が存在
- ステークホルダーが実際に触って判断できる
- デザイン修正は機能確認後なので、より適切な修正が可能
- 技術的制約が明確になった状態でデザイン開始
- 総工期が短縮される

## 実装のポイント

### 設計フェーズ
- ユーザー要件と機能要件を明確に定義
- データベーススキーマの基本構造を決定
- APIエンドポイントの一覧化
- 画面遷移図を簡潔にまとめる
- 詳細なビジュアルデザインは決めない（後工程で決定）

### AIコーディングフェーズ
- 設計書をベースに、Claude Codeで実装を実行
- フロントエンド（HTML/CSS/JavaScript）とバックエンド（サーバーロジック）を並行実装
- 動作確認可能な状態まで完成させる
- ユーザー認証、データ保存などの基本機能は実装
- エラーハンドリングも含める

### モック・デプロイ
- 開発環境またはステージング環境でデプロイ
- ステークホルダーが実際に操作できる状態にする
- 操作性やユーザーフローに関するフィードバックを収集
- 機能の過不足を確認

### デザインフェーズ
- 動作するシステムを見ながら、UIデザインを施す
- ブランドガイドラインに合わせたビジュアル統一
- ユーザビリティ検証を通じた微調整
- 視認性、操作感の改善
- レスポンシブデザイン対応

### 最終調整フェーズ
- デザイン反映後の動作確認
- パフォーマンス最適化
- セキュリティ監査
- ブラウザ互換性テスト
- 本番環境へのデプロイ

## AIツール選定のポイント

### Claude（チャット版）の活用場面
- システム全体の設計相談
- 技術的な課題の解決提案
- プロンプトの最適化
- フィードバック内容の言語化と方針決定

### Claude Code版の活用場面
- 実際のコード生成と実装
- ファイル操作と構造化
- 環境構築と依存関係管理
- バグ修正と機能追加

## 事例に見るメリット

### タレントマネジメントシステムの事例
- 企業内人材管理の要件を設計 → AIで実装 → 動作するモックでステークホルダーフィードバック取得 → 実際の業務フローに合わせた調整
- 従来であれば要件定義に3週間、デザインに2週間かかっていたが、AI活用で1週間に短縮

### 匿名褒めアプリの事例
- チーム内コミュニケーション向上を目標に設計 → AIで実装 → ユーザーが実際に使い体験 → 使いやすさに基づくUI改善
- 要望の言語化が格段に容易になり、修正ループが高速化

## 注意点と課題

- **品質保証**: AI生成コードの検証が必須。セキュリティやデータ整合性の確認を忘れずに
- **プロンプト設計**: 曖昧な要件指定ではAIも正確な実装ができない。設計段階での明確化が重要
- **スキルの変化**: エンジニアの役割が変わり、AIに指示を出す能力が求められるようになる
- **ツール依存**: 特定のAIツールへの過度な依存を避け、複数ツールの使い分けが推奨される
- **倫理的考慮**: AI生成コードの著作権やライセンスについての確認が必要

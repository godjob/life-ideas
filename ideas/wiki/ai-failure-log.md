# AIエージェント失敗ログと修正ナレッジ

## 概要

AIエージェントが犯したミスと その修正方法を体系的に記録・蓄積するプラクティスです。[AIオーケストレーター](ai-orchestrator-role.md)として人間が判断に集中するためには、AIの失敗パターンを事前に学習し、プロンプトやワークフローで防止することが不可欠です。このナレッジベースは複利的に効果を発揮し、時間とともにAIエージェントの精度が向上していきます。

## 主要な知見

- **失敗は資産**: AIエージェントの個別の失敗事例は、修正ナレッジとして蓄積すると、以後の同型ミスを防ぐための投資になる
- **複利効果**: 過去のミス→修正方法→プロンプト改善のサイクルを繰り返すことで、時間とともに指数関数的にエージェント精度が向上する
- **人間の判断に集中**: [AIオーケストレーター](ai-orchestrator-role.md)は「AIのコードを修正する」ではなく「AIが犯すミスのパターンを学ぶ」ことに集中すべき
- **プロンプト設計の重要性**: 失敗ログから抽出したパターンは、プロンプトエンジニアリングやワークフロー設計に直結する改善案になる
- **ドメイン知識の外部化**: AIが失敗した理由を言語化することで、暗黙知がドメイン知識として形式化される
- **Skillへの昇華**: 一つの難しいタスクをClaudeが成功するまで反復した実績は、[Skill System](skill-system.md)として再利用可能な資産に転換できる
- **経営判断の委任**：失敗したルール・制約は、[CLAUDE.md統治](claude-md-governance.md)により「止まるべきポイント」として経営方針に明文化され、それ以外の判断はAIに委任する。失敗は即座に組織の資産に変わる
- **高速反復による適応**: スペースXの「失敗をデータに変える速度」に学ぶと、予測不可能な環境下でも迅速に適応できる学習ループが競争優位性になる。[高速反復学習ループの運用設計](rapid-iteration-learning-loop-operational-design.md)により、従来の「ゼロ欠陥志向」から「高速反復による適応」へのシフトが必須になる

## 失敗ログの記録方法

### 記録すべき要素

1. **失敗事例**: AIが何を誤ったか（具体的な出力や判断ミス）
2. **発生コンテキスト**: どのようなタスク・入力で発生したか
3. **根本原因**: なぜそのミスが発生したのか（プロンプトの不備、学習データの偏り など）
4. **修正方法**: プロンプト改善、入力フォーマット統一、ワークフロー追加検証 など
5. **検証方法**: 修正後、同じミスが繰り返されないことを確認するテスト
6. **Skill化の判断**: 修正が安定した場合、[Skill System](skill-system.md)として組織化すべきかを評価
7. **ルール資産化**: 重大な失敗（金銭・削除など）であれば、[CLAUDE.md統治](claude-md-governance.md)で「止まるべきポイント」として明文化
8. **学習速度の評価**: 同一パターンの失敗が繰り返される場合は、[高速反復学習ループ](rapid-iteration-learning-loop-operational-design.md)の設計を見直し、フィードバックサイクルを短縮化すべきか検討

### 記録フォーマット例

```markdown
### [失敗タイトル]

**失敗内容**: 
- AIが何を誤ったか

**発生コンテキスト**: 
- タスク内容
- 入力例
- 当時のプロンプト版

**根本原因**:
- プロンプトの曖昧性 / 指示の漏れ
- AIの得意・不得意領域の誤認識
- 入力データのフォーマット不統一

**修正方法**:
- プロンプトの具体化・例示の追加
- ワークフローへの検証ステップ追加
- 事前処理・バリデーションの実装

**検証方法**:
- テストケースの設定
- 過去の失敗事例での再実行確認
- 類似ケースへの応用テスト

**反復速度の工夫**:
- フィードバックサイクルの短縮化
- テスト自動化による高速検証
- 段階的ロールアウトによる早期発見

**Skill化の判断**:
- [x] Skill System化が必要 / [ ] ドキュメント参照でOK

**ルール資産化の判断**:
- [x] CLAUDE.md統治に追加 / [ ] プロンプト改善で対応可能
```

## 失敗ログの活用サイクル

1. **ログ記録**: タスク実行時にAIの失敗を検知・記録する
2. **パターン分析**: 複数の失敗ログから共通パターンを抽出する
3. **プロンプト改善**: 抽出されたパターンに基づいてプロンプトを改訂する
4. **ワークフロー強化**: 必要に応じて検証ステップやルールを追加する
5. **高速反復サイクル**: [高速反復学習ループ](rapid-iteration-learning-loop-operational-design.md)に組み込み、カオス環境での適応能力を強化する
6. **継続監視**: 改善後の実行で同じミスが再発しないことを確認する
7. **資産化**: 安定した修正はSkillまたはルール資産として組織化する

## 失敗ログの優先度評価

### 高優先度（即座に対応すべき失敗）

- **金銭・法務に関わる誤判断**: 修正は経営判断に委任し、[CLAUDE.md統治](claude-md-governance.md)で「止まるべきポイント」として明文化
- **データ削除・破壊の危険**: ワークフローに確認ステップを必須追加
- **顧客・取引先への直接影響**: プロンプト改善と検証の二重化
- **継続的に繰り返される同一パターンの失敗**: 高速反復サイクルの導入か、根本的なアーキテクチャ見直しが必須

### 中優先度（計画的に改善すべき失敗）

- **品質低下のパターン**: プロンプト改善で対応、テストケース化
- **効率低下の原因**: ワークフロー設計の見直し、[高速反復学習ループ](rapid-iteration-learning-loop-operational-design.md)への組み込み
- **属人化リスク**: [Skill System](skill-system.md)化による標準化

### 低優先度（ナレッジとして記録すべき失敗）

- **稀な例外ケース**: ドメイン知識として蓄積
- **境界的な判断**: AIの得意・不得意領域の理解を深化

## 失敗をデータに変える速度：運用設計のポイント

スペースXの事例から学ぶ、失敗を競争優位性に変えるための運用設計：

- **迅速なフィードバック**: 失敗を検知してから修正・検証・デプロイまでのサイクルを短縮化する。従来の「計画→実装→テスト→本番」ではなく、「小単位で試行→学習→適応」の繰り返し
- **可視化と情報共有**: 失敗ログをチーム全体で参照可能にし、同じミスの再発を防ぐ。ナレッジの形式化がスピードアップの鍵
- **予測不可能性の受容**: ゼロ欠陥を目指すのではなく、「予測できない環境下でいかに迅速に学習・適応するか」を運用の中心に置く
- **学習ループの機械化**: [Hook設計パターン](hook-design-patterns-ai-workflow.md)やワークフロー自動化により、フィードバックサイクルの手動作業を最小化する

詳細は[高速反復学習ループの運用設計](rapid-iteration-learning-loop-operational-design.md)を参照。

## 実装のコツ

- **継続性**: 失敗ログは日々の業務に組み込む。都度記録する習慣をつける
- **チーム共有**: 失敗ログはチーム全体で参照・改善する共有資産として管理する
- **定期レビュー**: 週単位でパターンを分析し、プロンプト改善のスプリント化を検討
- **成功事例との対比**: 修正後の成功事例も記録し、何がうまくいったかを形式化する
- **反復速度の計測**: 失敗検知から修正完了までの時間を記録し、学習サイクルの短縮効果を可視化する

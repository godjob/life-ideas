# システム能力向上による『死に掛けたコード』：古い前提に基づく設計債務と定期的な仮説検証

## 概要

モデルやシステムの能力が向上すると、古い前提に基づいた設計が足かせになり、『死に掛けたコード』として機能し続ける。Anthropicの[Managed Agentsのインターフェース分離設計](managed-agents-interface-decoupling-design.md)に見られるように、能力向上による前提無効化に耐性を持つ設計が重要である。定期的に古い仮説を検証し、不要な複雑性を積極的に削除する仕組みがシステム寿命を延ばす。

## 主要な知見

- **古い前提の足かせ効果**: システム能力の向上により、かつて合理的だった保守的設計が非効率になる。複雑なエラーハンドリングや迂回ルートが、新しいモデルの実行能力向上により不要化する
- **『死に掛けたコード』の増殖**: オーバーヘッド監視とコード品質指標だけでなく、「その複雑性は本当に必要か」という定期的な仮説検証が必要。削除候補の記録と実験が重要
- **ペース配分の誤りと長期運用**: 長距離走における「最初から飛ばしすぎない」と同じく、エージェント設計での過度に保守的なアプローチ（コンテキスト不安由来）は後の改善可能性を奪う
- **インターフェース設計による耐性**: 実装詳細に依存しない抽象化により、能力向上時の設計変更コストを最小化できる。[Managed Agentsのインターフェース分離設計](managed-agents-interface-decoupling-design.md)がこの原則を体現
- **製造システムへの応用**: 製造現場の長期稼働プロセスも同じ問題に直面。定期的な工程改善と前提検証が、[製造業のAI即日適用パターン](manufacturing-ai-quick-wins.md)へのステップアップを可能にする

## 中核となる課題：古い前提の可視化と検証

システムが所有する「古い前提」は、しばしば実装の深部に埋め込まれている。これらは以下のパターンで現れる：

### パターン1：過度な保守性設計

新しいモデルの登場前、開発チームはコンテキスト長の制限や推論精度への懸念から、複雑なバリデーション層やフォールバック機構を多層に構築していた。これは当時は防衛的で賢明な設計だったが、モデル能力向上後は**オーバーヘッド**になる。

例：Claude 3 Opusの導入によるコンテキスト拡大後も、Claude 2時代向けに設計された入力分割ロジックが残存し、パフォーマンスを阻害している。

### パターン2：前提に基づく複雑な制御フロー

「エラーが発生しやすい工程」という仮説に基づいて設計された複雑な検証・再試行ロジックが、実装改善によって原因が解決された後も残存する。

### パターン3：キャッシュと事前計算の過剰設計

かつてのトークン制限下では、結果の事前キャッシュが戦略的に重要だった。現在の大規模コンテキスト環境では、その維持コストが利益を上回る。

## 定期的な仮説検証のフレームワーク

### Step 1：前提の明文化

古い設計決定ドキュメントから「この判断が成り立つには何が真である必要か」を抽出する。

```markdown
前提: 単一リクエストの応答遅延が>2秒になるリスクがある
→ 根拠: トークン数制限 (32K context)
→ 現状: 100K context環境
→ 検証対象: 事前キャッシュの必要性
```

### Step 2：能力向上の定量化

[AI予算管理とROI最適化](ai-budget-management-roi-optimization.md)の観点から、システム能力向上の具体指標を測定：

- モデル推論レイテンシーの改善
- トークン効率の向上
- エラー率の低下

### Step 3：複雑性とのコスト・ベネフィット分析

古い複雑性を保持するコストと、削除による影響を定量化：

- 保守負荷の削減量
- テストコードの簡潔化による開発速度向上
- 新機能追加時の干渉低減

### Step 4：段階的削除と監視

[マルチエージェントパイプラインのエラーハンドリングとチェックポイント](multi-agent-pipeline-error-handling-checkpoint.md)と同じく、重要な変更も段階的に進める。

```markdown
Phase 1: キャッシュロジックを並列実行（新流路）
Phase 2: トラフィック10%を新流路へ
Phase 3: トラフィック50%、メトリクス検証
Phase 4: 完全切り替え、旧ロジック削除
```

## 長期連続運用との関連性

[長期連続稼働AIエージェント設計](long-running-ai-agent-design-patterns.md)では、1ヶ月以上の自律運用を想定するが、その間にモデルの改善やシステム能力向上が発生する。古い前提に固執した実装は、以下のリスクを生じさせる：

1. **新機能の統合困難性**: 古い仮説の上に新機能を積み重ねると、複雑性が二次関数的に増加
2. **デバッグの長期化**: エラー診断時に古い防衛層を通過しなければならず、root causeが見えにくい
3. **監視メトリクスの曇り**: 古い複雑性が含まれたメトリクスでは、真の改善を検知しにくい

## 製造業への応用

[製造業のAI活用機会](manufacturing-ai-opportunities.md)を実現するにあたって、既存システムが「古い自動化前提」で構築されている場合が多い。例えば：

- FAX受信→手入力という前提で設計された承認フロー
- 人手確認が必須という前提での検査工程
- 電話応答を前提とした受発注プロセス

これらは[承認フロー簡素化と権限委譲設計](approval-workflow-simplification-competitive-advantage.md)や[AIエージェントの運用展開](ai-agent-operations.md)で改善されるべき対象だが、古い前提が深く根付いていると、改善案そのものが受け入れられない。

定期的な「この工程は本当に人手が必要か」という問い直しが、[Manufacturing Automation Cost disruption](manufacturing-automation-cost-disruption.md)の段階へ進む鍵となる。

## 組織的な検証メカニズム

単なる技術的改善ではなく、組織全体に「古い前提の検証」を習慣化させることが重要：

- **四半期ごとの『死に掛けたコード』レビュー**: 複雑性が高く、最近修正されていないコンポーネントを抽出
- **エンジニアの提案制度**: 「この実装は古い前提に基づいている」という指摘を積極的に評価
- **メトリクスダッシュボード**: オーバーヘッド率、テスト時間、新機能追加時間の推移を監視

[QMS様式のAIプロンプト統治](qms-style-ai-prompt-governance.md)と同じく、製造業の品質管理手法を、ソフトウェア設計の仮説検証に適用することで、体系的な改善サイクルが実現できる。

## 実践的なチェックリスト

- [ ] 過去12ヶ月以上修正されていないコンポーネントを特定したか
- [ ] その複雑性が、現在の要件に必要か検証したか
- [ ] 古い設計判断の根拠（前提）を文書化したか
- [ ] 能力向上によってその前提が無効化していないか確認したか
- [ ] 削除実験の監視メトリクスを事前に定義したか
- [ ] 段階的削除のロールバック計画を準備したか

# descriptionフィールドの最適化：トリガー条件と機能説明の書き方

## 概要

descriptionフィールドは、AIエージェントが自動的に適切なSkillを選択・実行するための最も重要なメタデータです。「何をするか」と「いつ使うか（トリガー条件）」の両方を明確に記述することで、自動トリガーの精度が大きく向上します。曖昧な説明ではAIが判断できず、実装者の意図通りに動作しません。

Skillは単なるmarkdownファイルではなく、[スクリプト・アセット・データを含むフォルダ](skill-md-specification.md)として認識すべき存在です。その中でdescriptionは「トリガー判定に使われる重要な信号」として機能します。

## 主要な知見

- **「何をするか」と「いつ使うか」の両立が必須**：機能説明だけでなく、どのような場面で使用するべきなのかのトリガー条件を必ず記述する。descriptionは「Skillの要約」ではなく、「このSkillをいつ呼ぶべきか」を明確にするセクション
- **ユーザーの実際の言い回しを含める**：「スプリントを計画して」「新規顧客をオンボードして」など、実ユーザーが実際に口にするフレーズを盛り込むことで自動トリガーの精度が向上
- **曖昧な表現は自動実行を妨げる**：「プロジェクトを手伝います」のような曖昧な説明では、AIが他の選択肢との区別ができず正しくトリガーされない
- **1024文字以内の簡潔性**：制限文字数内で必要情報を効果的に伝える工夫が必要
- **Gotchasセクションとの連携**：実装時にClaudeが実際に失敗した場面から作られた[Gotchasセクション](claude-code-skill-design-gotchas.md)の情報が最も信号密度が高く、descriptionの精密化に役立つ

## なぜdescriptionが重要なのか

[Skill System](skill-system.md)において、descriptionフィールドはSkillの「何か」を定義する最初の接点です。[Claude Code Agent Teams](claude-code-agent-teams.md)のようなマルチエージェント環境では、複数のSkillから最適な選択肢を自動判定する際に、このdescriptionの質が直接的に実行精度に影響します。

[AI失敗ログ](ai-failure-log.md)でよく記録される「エージェントが期待と異なるSkillを選択」というエラーの多くは、descriptionの不明確さが原因です。

## 効果的なdescriptionの書き方

### 1. トリガー条件を最優先に記述する

descriptionの冒頭は「いつ このSkillを呼ぶべきか」を明記します：

```
「ユーザーがスプリント計画の支援を求めたときに、チームメンバーのキャパシティを確認し、
タスクを適切に割り当てるためのSkill」
```

トリガー判定の信号としてAIが読み込むため、行動のタイミングが曖昧では役に立ちません。

### 2. ユーザーの言語を反映させる

実際のユーザーが言いそうなフレーズを含めることで、自然言語マッチングの精度が向上します。例えば：

**❌ 悪い例**：「プロジェクト管理支援」  
**✅ 良い例**：「『新しいプロジェクトを立ち上げて』『スプリントを計画して』『マイルストーンを追加して』といったリクエストに応じて、プロジェクト構造の初期設定と進捗管理のセットアップを行う」

### 3. 他のSkillとの区別を明確にする

同じドメイン内に複数のSkillがある場合、descriptionでそれぞれの違いと適用範囲を明示します：

```
【データ分析Skillとの違い】
- レポート生成Skill：既存データの可視化とプレゼンテーション形式の作成が目的
- データ分析Skill：生データからパターン発見と統計的分析が目的
```

### 4. 前提条件と制限事項を述べる

トリガー時に必要な条件があれば明記します：

```
「顧客データベースが事前に設定されている場合に限り、
新規顧客情報の自動オンボーディングを実行するSkill」
```

### 5. 負のトリガー条件も記述する

「このような場合は呼ばないでください」という条件もAIの判断を助けます：

```
「リアルタイム監視や緊急対応が必要な場合は使用しないこと。
バッチ処理向けのスケジュール実行専用Skill」
```

## 実装例

### 例1：スプリント計画Skill

```
【トリガー】ユーザーが「来週のスプリント計画を立てて」「チームのキャパシティに合わせてタスクを割り振って」といった依頼をしたとき

【機能】チームメンバーのスケジュールと過去のベロシティデータから、現スプリントに適切なタスク数を判定し、優先度別に割り当てる。Jiraと連携し、自動でエピック・ストーリー・タスクの階層構造を構築。

【使わない場合】既にスプリントが確定している場合、緊急のホットフィックス対応、個人のTODO管理
```

### 例2：バグ報告の自動分類Skill

```
【トリガー】新しいバグレポートが提出されたとき、または「このバグの優先度を判定して」というリクエスト

【機能】バグ説明文から再現手順を抽出し、影響範囲・重大度・顧客へのインパクトを自動分析。優先度（Critical/High/Medium/Low）を付与し、担当チームへ自動ルーティング。

【使わない場合】既に優先度が決まっているバグ、セキュリティ報告（別プロセス）、機能リクエスト
```

## 文字数最適化のポイント

descriptionは1024文字以内という制約があるため、以下のテクニックで情報密度を高めます：

- **段落区切りを活用**：改行とMarkdown記号で視認性を向上させ、同じ文字数でも情報が整理される
- **パターンマッチングの言葉を優先**：トリガー判定に直結する「ユーザーの言い回し」を優先度高く
- **詳細は他セクションへ**：Gotchasセクションやノートセクションで補足し、descriptionはコンパクトに

## Gotchasセクションとの関係

[Gotchasセクション](claude-code-skill-design-gotchas.md)は実装の落とし穴を記録するセクションですが、ここに記された内容は「AIが実際に失敗した場面」を表しています。このため、Gotchasから得られた知見をdescriptionに逆流させることで、より正確なトリガー条件が実現できます。

例えば、Gotchasに「顧客データが不完全な場合、オンボーディングフローが途中で止まる」と記されている場合、descriptionに「顧客の基本情報（メール・電話・住所）が揃っている場合のみ実行」と追記すべきです。

## チェックリスト

descriptionを完成させたら、以下の項目を確認します：

- [ ] トリガー条件が冒頭に明記されているか
- [ ] ユーザーの実際の言い回しが3つ以上含まれているか
- [ ] 他の類似Skillとの違いが明確か
- [ ] 使ってはいけない場面が記述されているか
- [ ] 1024文字以内に収まっているか
- [ ] 最新のGotchasセクションの内容が反映されているか
- [ ] 前提条件・制限事項が明記されているか

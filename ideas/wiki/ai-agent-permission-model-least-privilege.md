# AIエージェント権限モデル：最小権限原則とエアギャップアーキテクチャによるシステム保護

## 概要

AI開発ツールやエージェントの普及に伴い、セキュリティリスクが急増している。経営者・システム管理者が実装すべき重要な防御戦略は、エージェントに必要最小限の権限のみを付与する「最小権限原則」と、本番環境と開発環境を完全に分離する「エアギャップアーキテクチャ」である。これらの設計により、データロス、踏み台化、サイバー攻撃による全社規模侵害を防ぐことができる。

## 主要な知見

- **最小権限原則（Principle of Least Privilege）の実装**
  - AIエージェントには、その業務遂行に必要となるAPI権限のみを付与する設計が必須
  - 「全体アクセス可」という運用は、データロスと踏み台化のリスクを同時に高める
  - 権限スコープを機能単位で明確に定義し、定期的に監査する仕組みが競争力

- **エアギャップアーキテクチャによる生産システム保護**
  - 生産現場のシステム（工場制御ネットワーク）にAIエージェントが直接接続しない設計が必須
  - データ取得と指示出力のみに制限し、制御権の行使を人間が保持する
  - ネットワークを「参照レイヤー」「実行レイヤー」「制御レイヤー」に分離することで、侵害が全社規模に波及する事態を防止

- **分ける・残す・防ぐの3原則**
  - **分ける**：本番環境と開発環境、機能ごとの権限を完全に分離
  - **残す**：全ての操作・判断を監査ログに記録し、証跡を残す（バージョニング含む）
  - **防ぐ**：多層防御、権限検証、緊急時の復旧手段を整備

- **権限管理と監査ログの整備が効率化の前提**
  - 業務効率化の魅力に惹かれて導入すると、権限管理を後付けするハメになる
  - 監査ログ・緊急復旧手段（ロールバック機能、バージョニング）を先に整備してから本導入を進める判断が経営的に重要
  - 製造業では特に、システム管理者が「最小権限」と「エアギャップ」の2つを設計段階で徹底すること

## エージェント権限設計の実装パターン

AIエージェントの権限モデルは、[ai-security-non-engineer-governance-framework](ai-security-non-engineer-governance-framework.md)で述べた「分ける・残す・防ぐ」の原則を、具体的なシステムアーキテクチャレベルで実装する必要がある。

### 最小権限原則のスコープ設計

エージェントが保有すべき権限は、以下の粒度で明確に定義する必要がある：

1. **API権限の機能単位分割**
   - 在庫確認API → READ権限のみ
   - 発注APIを呼び出す場合 → 特定顧客・金額上限内のWRITE権限に限定
   - 本番システムへのアクセス → 禁止または管理者承認後のみ実行

2. **時間・リソースの制約付与**
   - 日次のAPI呼び出し回数上限
   - メモリ使用量の制限
   - 実行時間のタイムアウト設定

3. **定期的な権限監査**
   - 月次で権限スコープを見直し、不要な権限は即座に剥奪
   - 権限使用ログから「実際に使われていない権限」を特定する

### エアギャップアーキテクチャの設計

製造業における生産システムへのAI導入では、以下の3層分離アーキテクチャを採用することが推奨される：

```
┌─────────────────────────────────────┐
│ 参照レイヤー（AI可視化）             │
│ - 生産実績の読み取り（READ only）    │
│ - KPI監視・レポート生成              │
└─────────────────────────────────────┘
              ↓ データフロー（一方向）
┌─────────────────────────────────────┐
│ 判断レイヤー（AIが提案）             │
│ - 工程最適化の提案                   │
│ - 異常検知・アラート通知             │
└─────────────────────────────────────┘
              ↓ 提案通知（一方向）
┌─────────────────────────────────────┐
│ 制御レイヤー（人間が実行）           │
│ - 生産指示の発行（人間が承認）      │
│ - 装置制御・稼働管理                 │
└─────────────────────────────────────┘
```

この設計により：
- AIは生産データを読み取り、最適化案を提案できるが、制御権を持たない
- 人間（オペレーター・管理者）が最終判断し、実際の指令を発行する
- 攻撃者がAIエージェントを乗っ取った場合でも、制御レイヤーには直接アクセスできない

### 監査ログと復旧機能の統合設計

[ai-managed-service-operational-design](ai-managed-service-operational-design.md)で述べた「信頼性・監視・ロールバック機能の優先」は、権限管理においても同様に重要である：

1. **操作証跡の記録**
   - 全てのAPI呼び出し、データアクセス、判断プロセスをログに記録
   - ログの改ざん防止（外部ストレージへの送出、署名付き記録）

2. **緊急時の復旧手段**
   - エージェントが不正な権限使用をした場合、即座にロールバック可能な状態管理
   - バージョニング機構により、過去の正常な状態に戻す
   - 権限剥奪後も「何が起きたか」を追跡可能な設計

3. **権限使用の「タイムボックス化」**
   - 特定の期間内のみ拡張権限を付与し、自動的に剥奪する仕組み
   - 例：緊急対応用に1時間のみ本番アクセス権を付与

## 非エンジニア向けの権限管理チェックリスト

経営者・非エンジニアの管理職が、AIエージェント導入時に確認すべき項目：

1. **権限分離の確認**
   - [ ] 開発環境と本番環境が完全に分離されているか
   - [ ] エージェントの権限リストが文書化され、定期的に監査されているか
   - [ ] 不要な権限が発見された場合の「削除プロセス」が定義されているか

2. **監視・ロギング体制の確認**
   - [ ] 全ての操作がログに記録されるか
   - [ ] ログの改ざん防止措置が講じられているか
   - [ ] 管理者以外がログを削除・編集できないか

3. **復旧手段の整備**
   - [ ] 問題発生時にロールバックできる仕組みがあるか
   - [ ] バージョニング機構が実装されているか
   - [ ] 復旧に要する時間と手順がドキュメント化されているか

4. **定期的なセキュリティ監査の計画**
   - [ ] 月次で権限使用状況を見直す
   - [ ] 年次で全体アーキテクチャを検証する
   - [ ] インシデント発生時の対応フローが定義されているか

## AI時代の組織的なセキュリティ対応

[organizational-ai-adoption-acceleration-topdown-directive](organizational-ai-adoption-acceleration-topdown-directive.md)で述べた「トップダウン指示と自動追跡」は、セキュリティ領域においても同様に有効である。

最小権限原則の遵守を組織全体に浸透させるには：

1. **経営方針としての明文化**
   - 「全てのAIエージェントは最小権限原則に従う」を経営方針として発表
   - [ai-ceo-delegation-protocol](ai-ceo-delegation-protocol.md)の要領で、判断基準を明文化

2. **自動チェック機構の実装**
   - 権限付与時に自動的に「必要最小限か？」を検証するワークフロー
   - 定期的に権限スコープを分析し、不要な権限を自動検出する仕組み

3. **成功事例と失敗事例の共有**
   - [ai-failure-log](ai-failure-log.md)の要領で、権限設計の失敗から学ぶナレッジを蓄積
   - 部門横断的に「こういう権限委譲は失敗した」という事例を共有

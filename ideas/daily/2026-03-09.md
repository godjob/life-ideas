# 2026-03-09 アイデア・気づき

## スマイルカーブの終焉と「コ」の字型社会から得た気づき

出典: [安宅和人ブログ](https://kaz-ataka.hatenablog.com/entry/2026/02/10/132414)

### 1. 「中間工程の消滅」は製造業システム管理にも直撃する

- 調べる・比較する・叩き台をつくる・改善するという業務はAIが代替
- システム管理者として残るのは「何を導入するか判断する」「出てきた提案のダメ出し」
- 今やっていることがそのまま当てはまる → 今の役割は実は正しい方向にある

### 2. 「判断主権」という概念

- 与信モデル・採用アルゴリズム・医療優先順位などは「価値観の埋め込み」
- 社内システムも同じ: ERPやセキュリティポリシーの自動化は誰の価値観で動くか？
- **自社で判断基準を内製できない = 外部ベンダーに判断主権を渡している**
- 社内ポータル・セキュリティ運用のAI化は「内製できる範囲を広げる」視点で検討すべき

### 3. 「責任のクリーニング構造」への警戒

- AIの提案を承認するだけの「責任端子」化に自覚的になる
- 承認ボタンを押す前に「この判断の根拠を自分の言葉で説明できるか」を問う習慣

### 4. 子どもの教育への応用

- 中学生の娘・小学生の息子が社会に出る頃、中間工程的スキル（暗記・計算・整理）はAIが担う
- 育てるべきは「何がいい・悪い」「どこへ向かいたいか」という意志と価値観
- 答えを出す速さより、問いを立てる力を伸ばす

### 5. システム管理者として「コ」の字型のどこにいるか

- ディレクション側: ネットワーク設計方針・セキュリティポリシーの決定
- ダメ出し側: ベンダー提案のレビュー・障害対応の最終判断
- 中間（薄くなる側）: 定型的な設定変更・ログ確認・マニュアル作業 → 自動化推進が正解

## Anthropic「ClaudeのSkillを構築する完全ガイド」から得た気づき

出典: [Anthropic公式PDF](https://resources.anthropic.com/hubfs/The-Complete-Guide-to-Building-Skill-for-Claude.pdf)

### 1. Skillは「一度教えて、毎回使う」という発想の転換

- 会話のたびに同じことを説明するのは非効率
- SKILL.mdで一度仕様を書けば、すべての会話で自動的に適用される
- このlifeリポジトリのCLAUDE.mdと同じ考え方 → **既にSkillの思想を実践中**

### 2. プログレッシブディスクロージャーの設計哲学

- 第1レベル（YAMLフロントマター）: 軽量で常時読み込み、トリガー判断のみ
- 第2レベル（SKILL.md本文）: 必要時のみ展開
- 第3レベル（references/）: さらに詳細が必要なときのみ
- **教訓**: CLAUDE.mdも同様にコアだけを書き、詳細はリンク先に分離すると良い

### 3. MCP = キッチン設備、Skill = レシピ という比喩

- ツールへのアクセスだけでは不十分、使い方の知識が必要
- システム管理業務に応用: PowerShell/Ansibleスクリプト群（MCP）+ 運用手順書（Skill）の組み合わせが理想
- **社内ツール自動化の設計指針として活用できる**

### 4. 「descriptionに使うタイミングを明記する」原則

- 「何をするか」だけでなく「いつ使うか（トリガーフレーズ）」が必須
- 曖昧な説明（「プロジェクトを手伝います」）は機能しない
- Claude Code で `/git scrap` を呼ぶ場面の言語化をより具体的にすると効果的

### 5. skill-creatorを使った15〜30分での構築

- 上位2〜3のワークフローを把握していれば1セッションで構築可能
- 反復的な改善: エッジケース発見 → skill-creatorに持ち込む → 改善
- **次のアクション**: fuku-walkアプリの定型操作をSkill化できないか検討

## Microsoft Copilot Cowork から得た気づき

出典: [Microsoft 365 Blog](https://www.microsoft.com/en-us/microsoft-365/blog/2026/03/09/copilot-cowork-a-new-way-of-getting-work-done/)

### 1. 「チャットから実行」への転換が本格化

- Copilot Cowork は「答えを出す」から「仕事を完了させる」へのシフトを体現
- M365全体（Outlook・Teams・Excel）を横断して自律実行する点がClaude Codeと同じ構造
- 「意図 → プラン → バックグラウンド実行 → 承認」のループは、自分がClaude Codeで使っているワークフローと同じ思想

### 2. 社内システム管理への応用可能性

- カレンダー整理・会議資料生成・調達リサーチがAIで自動化される時代が来ている
- 製造業の社内でも「ベンダー提案のとりまとめ・会議準備・定例報告書の生成」はCoworkに近い用途で自動化できる
- **現実的なネクストステップ**: 社内でM365 Copilotが使える環境かどうかを確認する価値あり

### 3. AnthropicとMicrosoftの連携という事実

- Coworkの技術基盤にClaudeが組み込まれている（マルチモデル戦略の一環）
- Claude Codeを使っている自分が、M365 Copilotを使っても同じAIに乗れる可能性
- 「どのAIを使うか」より「どのワークフローに統合するか」が差別化になる時代

## Harness Engineeringベストプラクティスから得た気づき

出典: [逆瀬川ちゃんのブログ](https://nyosegawa.github.io/posts/harness-engineering-best-practices-2026/)

### 1. 「ハーネスへの投資は複利で効く」という本質

- リンタールール1つ追加 → 以降の全セッションでそのミスが防がれる
- テスト1つ追加 → 以降の全セッションで回帰を検出
- このlifeリポジトリでも同様：CLAUDE.mdの1行が毎回の会話に効く
- **教訓**: 今すぐ完璧にしなくていい。1セッションに1つ改善するだけで十分

### 2. 「リンターの仕事をLLMにさせない」原則

- CLAUDE.mdに「リンターを実行せよ」と書くことと、Hookで強制することの差は「ほぼ毎回」vs「例外なく毎回」
- コンテキストが消耗した47回目のセッションでリンターは忘れ去られる
- このlifeリポジトリへの応用：git commitのHookで誤字チェックや形式検証を自動化できる可能性

### 3. 「古いドキュメントはAIへの毒」という警告

- エージェントは「これは3ヶ月前のメモで今は古い」という直感がない
- リポジトリ内の古い説明文書・設計概要は最新の真実と区別できない
- **このlifeリポジトリへの適用**: CLAUDE.mdの古い記述を定期的にクリーンアップする習慣が重要
- ADR（Architecture Decision Records）の思想を取り入れる価値あり：「いつ・なぜそう決めたか」をタイムスタンプ付きで記録

### 4. エラーメッセージをそのまま修正指示にする発想

- カスタムリンターのエラーに「WHY / FIX / EXAMPLE」を書く → エージェントが自律修正
- エージェントはドキュメントは無視できるが、CIエラーは無視できない
- **応用**: CLAUDE.mdの指示を「〜してください」ではなく「〜しない場合はエラーとして扱う」形式に書き換えると強度が上がる

### 5. 最小実行可能ハーネス（MVH）の3点セット

1. フォーマッター自動実行（PostToolUse Hook）
2. テスト自動実行（Completion Gate Hook）
3. エラー注入（単なるブロックではなくフィードバックとして返す）

- 完璧なハーネスは不要。この3つから始めれば劇的に出力が安定する
- **このリポジトリでの現実的なMVH**: commitメッセージの日本語チェックHookを1つ追加するだけでも効果あり

## Claude Code週次ワークフロー活用術から得た気づき

出典: [ふぇね（山中裕貴）@X](https://x.com/0xfene/status/2030928836137783547)

### 1. 「日曜日を週の始まりにする」という発想の転換

- 日曜に来週の仕事を終わらせておくと月曜が超スムーズに始まる
- 振り返りをしないと「先週何やったっけ」から月曜が始まる → その無駄を排除する
- **このlifeリポジトリへの応用**: 日曜にgit scrapを実行してweeklyを自動生成するフローが既にある → 同じ思想

### 2. スキル化で「説明コスト」をゼロにする

- `/weekly-report`・`/mtg-prep` というスキルを作ることで、毎回説明する手間がなくなる
- スキルは「一度定義すれば何度でも使える」→ Harness Engineeringの複利効果と同じ
- **応用可能なスキル候補**: `/git-scrap`（既にある）、`/running-review`（月次走行記録の振り返り）、`/supplement-check`（補充タイミングの確認）

### 3. 「事後アクションの事前準備」という逆転の発想

- MTGの議事録や次のアクションを、MTG前に仕組みとして準備しておく
- 「MTGが終わったら何をするか」を事前に決めておくと実行速度が劇的に上がる
- **システム管理業務への応用**: 定例作業の手順をスキル化 → 毎回ゼロから考えなくてよくなる

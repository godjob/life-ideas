# 2026-03-09 アイデア・気づき

## スマイルカーブの終焉と「コ」の字型社会から得た気づき

出典: [安宅和人ブログ](https://kaz-ataka.hatenablog.com/entry/2026/02/10/132414)

### 1. 「中間工程の消滅」は製造業システム管理にも直撃する

- 調べる・比較する・叩き台をつくる・改善するという業務はAIが代替
- システム管理者として残るのは「何を導入するか判断する」「出てきた提案のダメ出し」
- 今やっていることがそのまま当てはまる → 今の役割は実は正しい方向にある

### 2. 「判断主権」という概念

- 与信モデル・採用アルゴリズム・医療優先順位などは「価値観の埋め込み」
- 社内システムも同じ: ERPやセキュリティポリシーの自動化は誰の価値観で動くか？
- **自社で判断基準を内製できない = 外部ベンダーに判断主権を渡している**
- 社内ポータル・セキュリティ運用のAI化は「内製できる範囲を広げる」視点で検討すべき

### 3. 「責任のクリーニング構造」への警戒

- AIの提案を承認するだけの「責任端子」化に自覚的になる
- 承認ボタンを押す前に「この判断の根拠を自分の言葉で説明できるか」を問う習慣

### 4. 子どもの教育への応用

- 中学生の娘・小学生の息子が社会に出る頃、中間工程的スキル（暗記・計算・整理）はAIが担う
- 育てるべきは「何がいい・悪い」「どこへ向かいたいか」という意志と価値観
- 答えを出す速さより、問いを立てる力を伸ばす

### 5. システム管理者として「コ」の字型のどこにいるか

- ディレクション側: ネットワーク設計方針・セキュリティポリシーの決定
- ダメ出し側: ベンダー提案のレビュー・障害対応の最終判断
- 中間（薄くなる側）: 定型的な設定変更・ログ確認・マニュアル作業 → 自動化推進が正解

## Anthropic「ClaudeのSkillを構築する完全ガイド」から得た気づき

出典: [Anthropic公式PDF](https://resources.anthropic.com/hubfs/The-Complete-Guide-to-Building-Skill-for-Claude.pdf)

### 1. Skillは「一度教えて、毎回使う」という発想の転換

- 会話のたびに同じことを説明するのは非効率
- SKILL.mdで一度仕様を書けば、すべての会話で自動的に適用される
- このlifeリポジトリのCLAUDE.mdと同じ考え方 → **既にSkillの思想を実践中**

### 2. プログレッシブディスクロージャーの設計哲学

- 第1レベル（YAMLフロントマター）: 軽量で常時読み込み、トリガー判断のみ
- 第2レベル（SKILL.md本文）: 必要時のみ展開
- 第3レベル（references/）: さらに詳細が必要なときのみ
- **教訓**: CLAUDE.mdも同様にコアだけを書き、詳細はリンク先に分離すると良い

### 3. MCP = キッチン設備、Skill = レシピ という比喩

- ツールへのアクセスだけでは不十分、使い方の知識が必要
- システム管理業務に応用: PowerShell/Ansibleスクリプト群（MCP）+ 運用手順書（Skill）の組み合わせが理想
- **社内ツール自動化の設計指針として活用できる**

### 4. 「descriptionに使うタイミングを明記する」原則

- 「何をするか」だけでなく「いつ使うか（トリガーフレーズ）」が必須
- 曖昧な説明（「プロジェクトを手伝います」）は機能しない
- Claude Code で `/git scrap` を呼ぶ場面の言語化をより具体的にすると効果的

### 5. skill-creatorを使った15〜30分での構築

- 上位2〜3のワークフローを把握していれば1セッションで構築可能
- 反復的な改善: エッジケース発見 → skill-creatorに持ち込む → 改善
- **次のアクション**: fuku-walkアプリの定型操作をSkill化できないか検討
